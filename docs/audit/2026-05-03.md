## Findings (28)

### P1 — Broken (4)

- [ ] (f1) `GET /api/admin/practice/[id]` returns **405** — only `PATCH` and `DELETE` are exported. Any client trying to fetch a single assessment by ID is blocked. The practice manager works around this by holding all assessments in list-fetched local state. `app/api/admin/practice/[id]/route.ts`. — owner agent
- [ ] (f2) `Competition.division` silently dropped on create. The field exists in the schema (`prisma/schema.prisma:507`) but `createSchema` in `app/api/admin/competitions/route.ts:24-32` never includes it, and `CompetitionForm` (`components/forms/competition-form.tsx`) has no division input. Always stored as `null`. — owner agent
- [ ] (f3) No publish/unpublish toggle in the competition UI. `app/dashboard/competitions/[id]/page.tsx:92` shows `isPublished` as a **display-only** badge. Competitions default to `isPublished: false` on creation. An owner using only the UI can never publish a competition — they have to call `PATCH /api/admin/competitions/[id]` directly. — owner agent
- [ ] (f4) `/api/member/hub` returns `approvedHours`, `pendingHours`, `expectedHours`, `totalHours` as `null` instead of `0` for members with zero hours. Member dashboard likely renders "—" instead of "0 hr". `app/api/member/hub/route.ts` (hour aggregation block). — member agent

### Track A — Engineering health

#### Slow with empirical backing (3)

- [ ] (f5) `/dashboard/members` — 620ms median, **275.6 KB payload**, jittery (+125% between runs). Confirmed: `app/dashboard/members/page.tsx` fetches all members server-side and ships them all to the client. With 50 seeded members it's already 275 KB; at 200 it would break. Almost certainly an N+1 around member-role aggregation. **Hypothesis:** add per-member-role include batching, paginate or virtualize the table.
- [ ] (f6) `/dashboard/competitions/[id]` — 534ms (OWNER), 526ms (MEMBER), consistent across runs. Both roles slow → not auth-overhead. Likely the competition-roster-manager (586 lines, see f10) is fetching roster + assignments + participants in a deep `include` chain.
- [ ] (f7) `/dashboard/practice` — 493ms (OWNER), 334ms (MEMBER). Jittery (+118-188ms vs prior run). Practice list builds attempts-by-assessment rollups; check whether `take: 20` from `lib/practice-assessments.ts` is still being honored after the query was rewritten.

#### Dead code (1)

- [ ] (f8) `lib/rate-limit.ts:155` — `export function rateLimitRetryAfterSeconds` is only used internally by `rateLimitErrorMessage` on line 160; zero external callers. Should drop the export. — owner agent

#### Refactor (2)

- [ ] (f9) `lib/practice-assessments.ts` — **1088 lines**, mixes five distinct concerns: Prisma query builders, type definitions/interfaces, business logic (scoring, progress), member-facing feed assembly, and admin list/CRUD helpers. Split candidates: `practice-queries.ts`, `practice-types.ts`, `practice-feed.ts`. — owner agent
- [ ] (f10) `app/dashboard/competitions/[id]/competition-roster-manager.tsx` — **586 lines**. Inline roster CRUD, assignment CRUD, participant CRUD, and three dialog open-states all in one component. Assignment sub-section (~250 lines) extractable as `CompetitionAssignmentPanel`. — owner agent

### Track B — Product

#### Missing functionality (8)

- [ ] (f11) **Bulk approve/reject for hours.** Admin queue can have 50+ entries (200 seeded, 50 pending). No multi-select / "Approve all" — every entry needs two individual clicks. `app/dashboard/hours/admin-hours-view.tsx` + `app/api/admin/hours/route.ts` (single-entry PATCH only). — owner agent
- [ ] (f12) **Approved/rejected history tab in admin hours.** Once an entry is approved/rejected it disappears from the queue (`app/dashboard/hours/page.tsx:22-24` filters to `status=PENDING`) and there's no "View all entries" view to audit past decisions. — owner agent
- [ ] (f13) **Member search/filter on the Members page.** `GET /api/admin/members` supports `?q=` search but `app/dashboard/members/page.tsx` ignores it: fetches all members and `MembersTable` has no filter input. Already annoying at 50; broken at 200. — owner agent
- [ ] (f14) **Competition-level division field.** Div B vs Div C is the most important metadata for an invitational. Field exists in schema and on `CompetitionRoster`, but is missing from the competition-level create/edit API and UI form. Connected to f2 (silent drop). — owner agent
- [ ] (f15) **Edit / withdraw own hour entries** (member). `/api/member/hours/route.ts` only exposes GET and POST. After hitting Submit there's no PATCH or DELETE — even before review. Common case: typo in description, wrong category. — member walk
- [ ] (f16) **Self-enroll in events** (member). `EventEnrollment` is admin-only (`/api/admin/events/[id]/enrollments` is the only writer). Members have no way to express interest in an event, so the entire `INTERESTED → TRYOUT_PENDING → ACTIVE` lifecycle described in the schema can only be admin-driven. Major friction for season setup. — member walk
- [ ] (f17) **Auto-populate or clarify default competition rosters.** Seeded member has zero competition assignments because the seed never populates `CompetitionRoster` membership for them. Same is true for real members in fresh seasons until rosters are built by hand. Consider seeding a "default roster" on competition creation, or much clearer empty-state copy. — member walk
- [ ] (f18) **Member-facing form submission API.** Members can `GET /api/member/forms` (lists required forms with their submission status) but there is no POST/PATCH endpoint to actually submit one (acknowledgement / file upload). The 90 seeded `FormSubmission` rows exist only because the seed inserts them directly. — member walk

#### UX rough (6)

- [ ] (f19) Admin hours view doesn't show submission date. `entry.submittedAt` is in the `Entry` interface (`admin-hours-view.tsx:26`) but is never rendered. Admins can't tell if a submission is from today or three weeks ago. — owner agent
- [ ] (f20) Hours `approvedBy` / `approvedAt` invisible in UI. After approval those fields are stored but never surfaced anywhere — members can't see when/who reviewed their entry. — owner agent
- [ ] (f22) Practice assessment part-type enum mismatch error. The only valid `AssessmentPartType` values are `SECTION` and `STATION`. The 400 response just says `"Invalid option: expected one of 'SECTION'|'STATION'"` with no context about which field is wrong. — owner agent
- [ ] (f23) **Hour entry has no autosave**. Practice attempts have a 120/minute autosave PATCH endpoint (good). The hour-entry form does not — long `description` text is lost if the tab closes mid-typing. Asymmetric UX between two member-side forms. — member walk
- [ ] (f24) No status filter on member's hours list. `GET /api/member/hours` returns all entries; no `?status=PENDING` filter. Members have to scroll the full list to find pending ones. — member walk
- [ ] (f25) Hub aggregates have no category-target progress display, despite `categoryTargets: []` being a placeholder in the response. — member walk

#### UI improvement (4)

- [ ] (f26) Hours admin queue card omits submission date and member email. Each card shows name and description but not when it was submitted or the member's email — hard to prioritize or cross-reference. `app/dashboard/hours/admin-hours-view.tsx:106-132`. — owner agent
- [ ] (f27) Competition list cards render raw enum values. `comp.type` shows as `"INVITATIONAL"` in the badge (`competitions/page.tsx:60`) instead of "Invitational". `TYPE_COLORS` map exists; a parallel `TYPE_LABELS` map is missing. — owner agent
- [ ] (f28) `"No canonical competition rosters yet"` empty state — copy is confusing to non-technical owners; "canonical" is jargon. `competition-roster-manager.tsx:425-427`. — owner agent
- [ ] (f29) Member's empty assigned-competitions state. When `competition.assignments` is empty, the page shows a default empty state. Without "Talk to your club admin to be assigned to a competition" copy, members may think the page is broken. — member walk
